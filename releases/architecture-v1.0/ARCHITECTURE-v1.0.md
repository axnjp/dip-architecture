# Domain Intelligence Platform — Architecture v1.0 Release Candidate

Status: `release_candidate_pending_approval_and_immutable_pin`

Classification: `Public Reference Architecture`

Date: `2026-07-20`

## Candidate authority

This candidate is assembled from the five approved, normalized current-source
packets in `foundations/`, approved under `PG0-E001` and P0-B46. The packets are
current implementation and publication authority under the approved exception;
they are not represented as an original conversation export.

Historical wording, exact original message locators, and the exact historical
enumeration of the 22 constitutional principles remain provenance limitations
accepted under `PG0-E001`. They do not alter the current approved architecture
or its implementation boundaries.

## 1. Platform purpose

DIP is a Governed Companion Intelligence Operating System around replaceable
foundation models. It provides governed memory, knowledge, intelligence,
workflows, review, orchestration, strategy, explanation, and auditability.

The initial implementation surface may be narrow, but the governing architecture
and trust boundaries remain intact.

## 2. Governing hierarchy

The model and platform hierarchy is:

**Foundation Model → AI Operating Profile → Domain Package → Workspace → Private Strategy**

The Foundation Model supplies reasoning capability. The AI Operating Profile
governs model behavior. The Domain Package supplies domain semantics. The
Workspace supplies operational context. Private Strategy contains private,
user-controlled strategic context and intellectual capital.

## 3. Reasoning and intelligence model

DIP keeps Memory, Knowledge, and Intelligence distinct:

- Memory: Historian and Event Ledger.
- Knowledge: Knowledge Graph and Knowledge Nodes.
- Intelligence: Findings, Claims, Recommendations, and the Decision Engine.

The Explanation Framework connects intelligence to human interaction. Evidence,
confidence, uncertainty, and decision-relevant explanation remain visible.

## 4. Governed operation

Authoritative artifacts follow:

**receive → validate → accept → version → propagate → observe**

Durable execution follows:

**Trigger → Workflow → Stage → Job → Attempt**

Execution is at-least-once with idempotent jobs. Refresh, replay, backfill, and
rebuild remain distinct operations. Review Service authority is preserved for
governed review and mutation.

ChatGPT and the DIP Operations Page are peer interfaces to the Review Service.
Recommendations, decisions, commands, and outcomes remain distinct:

**Recommendation → Decision Record → Command → Outcome**

## 5. First-domain scope and qualification boundary

Dynasty fantasy football is the first Domain Package. Trevecca’s Finest is the
first operational Workspace. Master Minds remains a viable second Workspace
for portability, multi-Workspace, and League State Intelligence testing.

The completed Master Minds draft does not validate startup-draft ingestion,
draft-state behavior, or draft-to-normal-operation lifecycle behavior. That
validation is deferred to a suitable future Workspace or controlled draft-state
fixture. Sleeper remains a read-only external source for the first vertical
slice, League State Intelligence.

External market values remain a prerequisite for value-dependent recommendations.
The initial deployment posture is private and low-cost while preserving the
long-term architecture.

## 6. AI operating profile and constitutional boundary

The AI operates as a Strategic Intelligence Partner. It is evidence-first,
confidence-calibrated, explanation-oriented, aligned to user goals without
automatic agreement, bounded in initiative, and portable across foundation
models.

Memory, context, Workspace, Private Strategy, MCP interaction, interface parity,
attribution, failure handling, safe degradation, adaptation, and deployment
conformance remain governed capabilities.

The DIP AI Constitution governs these duties. Its normalized 22-principle
reconstruction is approved as current authority under `PG0-E001`; it is not
claimed to reproduce unavailable historical titles, sequence, or wording.

## 7. Security and disclosure boundary

The publication model separates:

1. Public Reference Architecture;
2. Restricted Architecture;
3. Private Authority; and
4. Secrets.

The public candidate uses an explicit allowlist and contains no restricted
source bodies, private authority data, active private configuration, or secret
values. Private Strategy is described only as an abstract capability. Public
documentation does not establish whether any real Private Strategy instance
exists.

## 8. Implementation and release boundary

This candidate authorizes review of Architecture v1.0. It does not authorize
runtime/platform coding. Coding requires, in order:

1. Architecture v1.0 approval;
2. immutable release pinning;
3. formal PG0 passage against that pin; and
4. explicit IKG0 authorization.

The first implementation remains subject to the approved implementation plan,
ADR-IMP-001 through ADR-IMP-029, P0–P6 publication controls, and the current
scope/deferral boundaries.

## 9. Candidate decision

The assembled candidate is semantically ready for approval under the accepted
provenance exception. The candidate is not yet authoritative as a released
Architecture v1.0 edition because no release approval, release tag, or immutable
release pin exists.
